适用场景：

  • 由于同事离职，将工作交接给自己。
  • 由于自己离职，将工作交接给同事。
  • 由于项目变动，将工作交给其他项目组。

项目资料点：总

  • 将开发人员最后修改的代码提交。

  • 前端对接人

  • 后端对接人
  • 后端 git
  • 前端 git 地址
  • git 分支
  • 开通 GIT 的权限。
  • 了解代码规范，以及其他规范文档
  • 本地开发环境配置
  • 在自己本地电脑上配置环境，将项目在自己机器上运行成功。
  • 确认是否有其他的扩展，如需账号、端口，记得做记录。
  • 确定项目对外 API。
  • 了解上线部署流程。
  • 测试环境地址
  • 正式环境地址
  • 接口文档
  • 设计稿地址
  • 原型
  • TAPD 需求
  • jekins
  • TAPD bug
  • 项目难点
  • 注意事项
  • 把项目代码和页面逻辑面对面或者语音说明一遍

测试环境

  • 开通测试账号。

预上线环境

  • 开通预上线环境账号。
  • 确定预上线环境的域名地址，是否需要指定的 Host 等等。
  • 正式环境
    开通正式环境账号。

对接人相关注意点：

  • 确定测试对接人。
  • 确定产品对接人。
  • 确定项目跨部门对接人。
  • 确定后端负责人
  • 确定前端参与人员

文档相关注意点：

  • 确定每一期的产品需求文档地址
  • 确定每一期的测试覆盖文档，单元测试文档，遗留 Bug。
  • 确定每一期的产品人员，开发人员，测试人员。
  • 确定代码中的坑、梳理过的流程图、设计框架图。
  • 确定每一期的开始开发时间，提测时间，上线时间。
  • 了解相关程序风险，遗留问题等。
  • 文档越多越好，未必会看，但留着备用总是好的。
  • 最好能邀请一个会议进行交接，叫上相关产品、开发、测试。

职业道德：

  • 不要带入个人情绪，个人恩怨。
  • 要实现毫无保留的完成项目交接。
  • 组织一个会议进行项目交接，对于交接人员提出的问题，仔细逐一讲解、解答。
  • 交接单以交接人员签字确认为准，不要浮于形式，要走心。

备注：

  • 最好能在员工提离职的时候就开始项目交接，而不是员工走的时候再做交接。
  • 离职要提前与上级领导沟通，给领导留出找对接人的时间。
  • 交接过程中遇到有疑问的地方，一定要确认清楚，做好记录。
