前端工作经验总结

HYTD
2020-10-30
博客

保密制定

  • 公司产品代码，项目代码，资料，不允许放到 github 等开源网站，一经发现严肃处理，并且保留追究其法律责任的权利

沟通例子：

对甲方客户沟通问题

  • 不要跟客户犟嘴。赢了也是输啦。目的是让他们帮忙解决问题。
  • 客户要求超出需求，及时上报，理性沟通。不要直接搞，或者直接拒绝
  • 客户给的项目资料要及时整理同步和同步，不要重复给客户要
  • 需要客户帮忙添加的，例如开发权限之类的，要先内部收集名单，不要重复让客户帮忙添加

沟通技巧：

不要说

  • 你理解啦吗？
  • 你清楚啦吗？
  • 你能明白吗？

改成

  • 我说清楚了吗？

带项目常见问题

  • 需求，是最重要的，一定要明确了解需求
    接到新需求之前，需要详细了解需求之后，再考虑是否能实现，实现的成本是否过大，如果有疑问需要跟产品再进行敲定具体需求，再进行具体开发
    开发期间需要及时沟通需求，不能对需求有任何不懂或者疑问的地方，否则很可能会导致重新代码甚至项目延期。

  • 项目规范：有 eslint 定义的规范一定要遵守。 原则上一定要有 eslint 文件定义规范 ， 没有的 eslint 文件的向上请示。（大家养成规范写法。 习惯了就不觉得费劲了）

  • 项目脚手架 项目基础脚手架按照客户要求。没有的一定要向上请示。切勿自己盲目使用，导致后期迁移成本过高

  • 内部沟通：尽量在群里沟通，拉对应的负责人。除非必要，不要私聊

  • 合理安排任务：项目负责人不要把自己开发任务排的过于饱满，多把精力放到项目把控上，包括项目 进度，风险

  • 暴露风险：负责人要有风险意识，项目负责人了解风险后，要及时同步到项目经理。

  • 同步进度：开发进度，及时同步跟项目经理，测试，产品

  • 资料收集：项目资料及时整理并发布到 tapd。

  • 规划： 对项目进度，人力，节奏要有大致规划，不要卡卡盲目的进行。

  • 早会晚会：新人带项目没有经验，你只要记得每天开早会晚会即可。重要的是把控和了解的进度和风险

  • 客户管理：

  • 沟通原则：跟客户沟通要温和，不跟客户犟嘴，不然你就算赢了，也是输啦。 目的让客户帮忙解决问题

  • 沟通节奏：客户不合理的需求，最好能带下客户节奏，尽量能给出解决方案。给客户建议。

  • 责任心： 给予自己的任务认真思考，提前完成，别延后

  • 细心： 出现问题后，先找自己的原因，不要先入为主认为是别人的问题，沟通的时候，也要说你看是不是我这边的问题（尤其跟甲方沟通的时候）

  • 别推延： 一定要做的事情一定提前做，不要非得被别人 diss 啦还得做

开发常见问题

  • 责任心： 给予自己的任务认真思考，提前完成，别延后
  • 暴露风险：有风险的及时沟通，避免耽误进度
  • 紧急的问题，如果自己 20 分钟以内解决不了问题，优先请教同事，解决紧急问题之后，找时间自己去消化该问题。
    一般的问题可以适量给自己多一点时间，如果没有思路一定要及时询问其他人

不足：

  • 前端的内部分享做的不够
    尽量抽出时间做技术分享
  • 前端的内部技术问题解决方案没有技术同步到其他人
    整理出对应的文档
  • 项目风险无法及时暴露
    做项目规划，规划计划，设计稿，静态页面，模拟接口，交互，真实接口，测试，提测的每个时间点！ 能明确看出来问题出在哪里。

自身的一些问题

  • 参与项目少，了解项目需求少
    多参于项目
  • 只关注组长，对成都这边基础员工了解不多
    定期跟基础员工沟通
  • 精力过于分散，工作切换频繁，没有产出
    优化工作流程，减少风险

欢迎强总 boss 指点工作中的不足
